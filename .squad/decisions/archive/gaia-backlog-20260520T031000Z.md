## Summary

The implemented app substantially matches the approved onboarding → main flow on structure (sticky disclaimer, no-default Fitzpatrick, location rationale before prompt, "Apple Weather" attribution, sunscreen 2-hour cap, gauge inside hero card, circular burn-risk ring, compact Location + SPF chip row WIP). Three material gaps stand out: (1) **the user-facing privacy disclosure copy and the README still claim "skin type and SPF stay in app memory only"**, which contradicts the ratified persistence decision Kwame already shipped — a Plunder-grade truthfulness gap and a launch blocker; (2) **the UI test suite on the in-flight branch is not green** — a clean re-run failed with simulator-process restarts around `testBurnRiskGaugeShellExistsWhenNoEstimate`, so the WIP cannot be merged as-is; (3) the README "User scenarios captured" still lists SPF "None" (removed by ratified decision) and claims SPF/skin type don't persist (they do). Everything else is polish or scope deferred to v1.1.

## Backlog (prioritized)

### WI-1: Bring privacy disclosure copy in line with ratified persistence

- **Priority:** P0
- **Owner:** Kwame
- **Reviewer:** Plunder
- **Type:** fix (legal/compliance truthfulness)
- **Source of gap:** D-2026-05-19 `gaia-preference-persistence.md` + `kwame-persist-user-preferences.md` were ratified and the implementation persists `selectedSkinType` + `selectedSPF` + `locationRationaleAcknowledged` via `@AppStorage` (`UserPreferenceStorage` in `UVBurnTimerSession.swift`; `RootView` `persist(skinType:)` / `persist(spf:)`). Yet `ProductCopy.aboutPrivacy` still reads *"Skin type and SPF stay in app memory only"* and `ProductCopy.cacheRetentionLine` still reads *"does not save UV values, burn estimates, skin type, or SPF"*. These strings are surfaced in Settings > Privacy and AboutView > Privacy. This is a material misrepresentation of the app's actual data behavior — a Plunder pre-submit flag #6 violation.
- **Acceptance criteria:**
  - `ProductCopy.aboutPrivacy` truthfully states that skin type and SPF are stored locally on device (UserDefaults plist, sandboxed, not leaving device, not in HealthKit), and that disclaimer acknowledgement remains transient (cold-launch only).
  - `ProductCopy.cacheRetentionLine` truthfully states what IS stored on device: last rounded coordinate, skin type, SPF, location-rationale acknowledgement — and what is NOT stored: UV values, burn estimates, raw GPS, photosensitizer status, any health data.
  - New copy still contains the four invariants Plunder needs: "Apple Weather", "rounded coordinates", "approximate location", "no accounts / no analytics / no ads / no crash SDKs / no third-party tracking".
  - Existing Settings > Privacy "Clear saved location" button stays present and still clears only the rounded coordinate.
  - No HealthKit, no Keychain, no remote sync (verified by `appSourcesAvoidProhibitedIntegrations`).
- **Test plan (TDD):** Update FIRST, then change copy:
  - `app/Tests/UVBurnTimerCoreTests/BurnTimeCalculatorTests.swift` → `aboutPrivacyCopyDescribesRoundedCoordinatesToAppleWeather` (line 508): replace assertion `contains("app memory only")` with `contains("stored locally on this device")` (or equivalent truthful phrase) AND add assertion that the copy mentions skin type + SPF persistence.
  - Same file → `approvedMainScreenSafetyCopyIsCaptured` (line 352–393): update `cacheRetentionLine` assertions at lines 365–366 to require the new truthful wording.
  - Add a NEW test `aboutPrivacyCopyMatchesActualPersistenceBehavior` that asserts the copy mentions every key persisted by `UserPreferenceStorage` (`selectedSkinTypeKey`, `selectedSPFKey`, `locationRationaleAcknowledgedKey`, `lastRoundedCoordinate`) in user-readable form.
- **Estimated touched files:**
  - `app/Sources/UVBurnTimerCore/ProductCopy.swift`
  - `app/Tests/UVBurnTimerCoreTests/BurnTimeCalculatorTests.swift`
  - `README.md` (scenario 10 + Privacy guardrails section — see also WI-2)

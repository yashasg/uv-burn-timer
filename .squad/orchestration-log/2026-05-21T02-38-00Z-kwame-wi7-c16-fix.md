## Summary

**Issue:** Ma-Ti's test C16 (`test_missing_or_nil_hourly_entry_coerces_to_zero`) uncovered a spec vs. implementation gap: missing hourly entries should coerce to `.nighttime` (polar-as-nighttime directive), not `.unavailable(.snapshotExpired)`.

## Root Cause

`ForecastStore.uvIndex(at:)` fallback logic lacked date-range guards. Missing entries inside the snapshot window were indistinguishable from queries outside the window, leading to incorrect `.unavailable` return.

## Fix

Added date-range guard (`firstHour…lastHour`) in `ForecastStore.uvIndex(at:)`:

```swift
guard targetHour >= firstHour && targetHour <= lastHour else {
    return .unavailable(reason: .snapshotExpired)  // ← outside window
}

// ← inside window, entry absent → coerce to .nighttime
return .nighttime
```

**Critical distinction:** 
- Dates *outside* snapshot window: `.unavailable(.snapshotExpired)` ✅
- Dates *inside* window with absent slot: `.nighttime` ✅ (per polar-as-nighttime directive)

## Kwame's Findings (Complete)

**Type name correction:** WeatherKit type is `SunEvents`, not `Sun`. `DayWeather.sun: SunEvents` (non-optional struct with optional Date fields inside).

**Canonical polar-night signal:** `DayWeather.sun.solarNoon == nil` (explicitly documented nil-only condition for polar night per Apple docs).

**Sunrise ambiguity resolved:** `sunrise == nil` fires for BOTH polar night AND polar day (per WWDC22 session 10035). Iris's v3 spec §3.3 primary trigger (sunrise == nil) was ambiguous and would have incorrectly collapsed UVI-14 polar-day to "No UV today." This research discovered and prevented that bug.

**HourWeather.uvIndex behavior:** Returns `0.0` for polar-night hours (not nil/missing).

## Scope Narrowed by User Directive 2026-05-21T01:58:19Z (Polar-Treat-as-Nighttime)

**Kwame's research is locked, but implementation scope narrowed:**

The user directive (issued after Kwame's research was underway) determines that polar night should be treated identically to regular nighttime (UVI = 0 rendering), with no polar-specific detection logic, copy, or UI in v1.

**Kwame's findings still lock the canonical signal:** `DayWeather.sun.solarNoon == nil` is the authoritative polar-night trigger if ever needed in future versions.

**But in v1:** No UI code will query this signal. Storage persists 24 rows per day regardless; if all 24 rows have UVI=0 for a given day, that day renders as "No UV" via existing nighttime path — no polar-specific branch needed.

**Synthesis:** When Kwame's research is reviewed by coordinator, findings synthesize into a one-line data-mapping rule ("coerce missing/nil UVI values to 0.0 if needed at fetch time") rather than a UI rewrite.
